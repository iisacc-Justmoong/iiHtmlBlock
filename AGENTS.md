# 저장소 지침

## 프로젝트 구조와 모듈 구성

이 저장소는 CMake로 `iiHtmlBlock` Qt6.8.3 + C++20 공유 라이브러리를 빌드한다. 현재 공개
진입점은 저장소 루트의 `iiHtmlBlock.h`와 `iiHtmlBlock.cpp`이다. 파서 관련 모듈은
`Src/Parser/` 아래에 둔다. 새 파서 코드를 루트에 두지 말고 해당 모듈 디렉터리에
배치한다. `Test/`는 빌드 및 실행 테스트 케이스를 위한 위치다. 디렉터리 구조는 앱
아키텍처의 청사진으로 취급하고, 새 모듈은 이 구조와 맞을 때만 추가한다. 빌드
디렉터리는 반드시 `build/`만 사용한다.

iiXml을 이용하여 Xml 태그를 받은 후 이를 HTML로 변환하여 조작하는 API를 제공하는 데 목표가 있다.


## 빌드, 테스트, 개발 명령

- `cmake -S . -B build`: 필수 빌드 디렉터리인 `build/`에 프로젝트를 구성한다.
- `cmake --build build`: `iiHtmlBlock` 공유 라이브러리를 컴파일한다.
- `ctest --test-dir build --output-on-failure`: CTest 대상이 등록된 뒤 테스트를
  실행한다.

새 작업에서 `cmake-build-debug/` 같은 대체 빌드 폴더를 만들지 않는다. 명령이
실패하면 우회하지 말고 근본 원인을 수정한다.

## 코딩 스타일과 명명 규칙

C++20을 사용하고 헤더는 대응하는 `.cpp` 파일과 함께 유지한다. 기존 클래스는
`GetHTML`, `GetTagInfo`처럼 PascalCase를 사용하므로 새 타입도 이 스타일을 따른다.
모듈명은 `Src/<Module>/` 형태로 명확하게 작성한다. 새 내부 헤더에는 기존 공개
헤더 관례와 충돌하지 않는 한 `#pragma once`를 선호한다. 모든 객체는 `QObject`를****
사용한다. 들여쓰기는 네 칸을 사용하고, 함수와 클래스의 여는 중괄호는 같은 줄에
둔다. 주석은 명확하지 않은 로직을 설명할 때만 짧게 작성한다.

## 테스트 지침

동작 변경은 반드시 `Test/` 아래 테스트와 문서 갱신을 함께 포함해야 한다.
`ctest`에 의존하기 전에 CMake에 `enable_testing()`과 `add_test()`로 테스트 대상을
등록한다. 테스트 이름은 모듈과 동작을 드러내게 작성한다. 예:
`Parser_GetTagInfo_ParsesAttributes`. 새 파서 동작에는 빌드 검증과 실행 시점 단언을
모두 포함한다.

## 커밋과 풀 리퀘스트 지침

현재 이 디렉터리는 Git 저장소로 초기화되어 있지 않아 로컬 커밋 관례가 없다.
커밋 제목은 `Add parser tag extraction tests`처럼 간결한 명령형 문장으로 작성한다.
풀 리퀘스트에는 변경 설명, 실행한 빌드 및 테스트 명령, 관련 이슈를 포함한다. UI
변경이 있을 때만 스크린샷을 첨부한다.
